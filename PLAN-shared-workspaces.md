# Implementation Plan: Shared Workspaces

## Overview

This plan adds shared workspaces to the octopus bot platform. A workspace is a
host directory that multiple bots can share, enabling collaborative work on the
same codebase. Workspaces are a deployment-time concept managed by `./octopus` —
no new runtime model is needed in `app/`. The existing `ProjectBinding`
abstraction (name, root\_dir, extra\_dirs, file\_policy, model\_profile) is the
right runtime model; workspaces just generate the correct `BOT_PROJECTS` values
and Docker volume mounts.

## Architectural Decisions

1. **Workspace is a deployment concept, not a runtime concept.** No new runtime
   model in `app/`. The existing `ProjectBinding` is the right abstraction.
   Workspaces just generate the right `BOT_PROJECTS` values.
2. **Separate workspace.env file per bot** — machine-owned, referenced from
   `docker-compose.workspace.yml` via service-level `env_file:` (and optional
   CLI `--env-file` only if the override YAML needs interpolation). Bot `.env`
   stays operator-owned.
3. **One bind mount at repo root, subpath projects** — prefer one `-v` mount
   and multiple `BOT_PROJECTS` entries for subpaths. Separate mounts only when
   security isolation requires it.
4. **No file locking** — git + delegation + human merge is the coordination
   contract.
5. **Workspace labels via existing tags** — `workspace:<slug>` tag on
   heartbeat, discoverable via existing `AgentDiscoveryQuery.tags`. No new
   registry entity.
6. **Membership is per logical bot, not per process** — scales naturally to
   shared runtime mode with worker replicas.

### Security boundary

A workspace bind mount gives every member bot full read access to the entire
mounted tree (and write access in `rw` mode). Do not mount directories
containing secrets you would not expose to every member bot. Use subpath
`BOT_PROJECTS` entries for internal directory splitting — these control the
provider CLI's working directory, but the container process can still reach the
full mount. For hard isolation within a tree, use separate workspace mounts at
disjoint subpaths.

Provider auth stays in `.deploy/provider-auth/<provider>/`, shared only by
provider policy. Workspace membership never widens credential scope.

## Data Model

### Filesystem layout

```
.deploy/
  workspaces/
    <workspace-slug>/
      workspace.conf          # WORKSPACE_ROOT, WORKSPACE_MOUNT, WORKSPACE_MODE
      members.txt             # one bot slug per line, source of truth
  bots/
    <bot-slug>/
      .env                    # operator-owned, unchanged
      workspace.env           # machine-generated, workspace project + tag additions
      docker-compose.workspace.yml          # local mode: bot + bot-provider only
      docker-compose.workspace-shared.yml   # shared mode: all 4 services
```

### workspace.conf format

```
WORKSPACE_ROOT=/Users/tinker/projects/shared-repo
WORKSPACE_MOUNT=/workspace/shared-repo
WORKSPACE_MODE=rw
```

- `WORKSPACE_ROOT` — absolute host path to the shared directory
- `WORKSPACE_MOUNT` — container mount point (always `/workspace/<name>`)
- `WORKSPACE_MODE` — `rw` (default) or `ro`

### members.txt format

```
my-bot
work-bot
```

One bot slug per line. Empty lines and `#` comments ignored. Single source of
truth for workspace membership.

### workspace.env format (generated per bot)

```
# Machine-generated by ./octopus workspace — do not edit.
# Regenerated on every workspace mutation.
BOT_PROJECTS=shared-repo:/workspace/shared-repo|edit
BOT_AGENT_TAGS=workspace:shared-repo
```

When a bot belongs to multiple workspaces, entries are comma-separated. The
generation step reads existing values from the bot's `.env`, appends workspace
entries, deduplicates by name, and writes the merged result. **Tie-break rule:**
workspace-generated project entries win over same-name entries from `.env`.

When a bot belongs to multiple workspaces, mount path uniqueness is guaranteed
by the `/workspace/<slug>` convention — each workspace slug is unique, so
mount paths cannot collide.

### Compose overrides (generated per bot, two files)

**`docker-compose.workspace.yml`** — local mode (used by `bot_compose`):

```yaml
services:
  bot:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
  bot-provider:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
```

Only `bot` and `bot-provider` — these are defined in `docker-compose.yml`.
Including `bot-webhook` or `bot-worker` here would create image-less services
and break the compose project in local mode.

**`docker-compose.workspace-shared.yml`** — shared mode (used by
`bot_shared_compose`):

```yaml
services:
  bot:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
  bot-provider:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
  bot-webhook:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
  bot-worker:
    env_file:
      - .deploy/bots/my-bot/workspace.env
    volumes:
      - /Users/tinker/projects/shared-repo:/workspace/shared-repo:rw
```

All four services. `bot-webhook` and `bot-worker` are defined in
`docker-compose.shared.yml` (merged before this override in the `-f` chain).

Paths in the override are relative to the compose project directory (set via
`--project-directory .` in `bot_compose()` / `bot_shared_compose()`).

**Critical:** `BOT_PROJECTS` and `BOT_AGENT_TAGS` must reach the container
process. Compose CLI `--env-file` feeds YAML interpolation (`${VAR}`), not
the container environment. The compose override must attach workspace.env as a
service-level `env_file:` entry on every affected service. Compose merges
`env_file:` lists; entries from later override files are appended, and
duplicate keys resolve last-wins within the merged list. Since the base
compose loads `.env` via `env_file: ${BOT_ENV_FILE}`, and the workspace
override appends workspace.env after it, workspace values win for overlapping
keys.

Compose also natively merges volume lists from override files. This avoids
shell-level `-v` flag injection and follows the existing pattern of
`docker-compose.shared.yml` as an override.

**Verification:** Confirm `env_file` merge semantics with the target Compose
version before implementation. Record the confirmed behavior in a comment or
test.

### Functions to add to `octopus`

**`cmd_workspace()`** — top-level dispatcher (pattern follows `cmd_registry()`):

| Subcommand | Behavior |
|---|---|
| `create <name> <host-path>` | Create workspace config, validate path, create empty members.txt |
| `remove <name>` | Remove workspace dir, regenerate workspace.env for all former members |
| `add-bot <workspace> <bot-slug>` | Append to members.txt, regenerate workspace.env for that bot |
| `remove-bot <workspace> <bot-slug>` | Remove from members.txt, regenerate workspace.env for that bot |
| `status [workspace]` | Show workspaces, members, effective BOT\_PROJECTS |
| `verify [workspace]` | Dedicated validation (see Phase 4) |

Validation rules for `workspace create`:
- Name: valid slug, non-empty, ≤32 chars
- Host path: absolute, exists, is a directory
- Reject if workspace already exists

All operations idempotent: creating existing workspace fails cleanly, adding
existing member is a no-op, removing non-member is a no-op.

Add `workspace` to `main()` case statement and `cmd_help()` output.

### Test strategy

- Create with valid/invalid paths
- Create with duplicate names
- Add-bot / remove-bot membership tracking
- Idempotency of all operations
- Status output format

## Phase 2: Workspace.env Generation

### Files to modify

| File | Change |
|---|---|
| `octopus` | Add `regenerate_bot_workspace_env()` function |

### Core generation function: `regenerate_bot_workspace_env()`

Logic:
1. Input: bot slug
2. Find all workspaces the bot belongs to (`bot_workspace_memberships`)
3. If none: remove workspace.env and docker-compose.workspace.yml, return
4. Read existing `BOT_PROJECTS` from bot `.env`
5. Read existing `BOT_AGENT_TAGS` from bot `.env`
6. For each workspace membership:
   - Read workspace.conf values
   - Build project entry: `<workspace-name>:<mount-path>|<policy>` where
     policy is `edit` for rw, `inspect` for ro
   - Build tag: `workspace:<workspace-slug>`
   - Build volume line: `<host-path>:<mount-path>:<mode>`
7. Merge: existing BOT\_PROJECTS + workspace entries (deduplicate by name;
   workspace wins on name collision)
8. Merge: existing BOT\_AGENT\_TAGS + workspace tags (deduplicate)
9. Write `.deploy/bots/<slug>/workspace.env` (permissions 0600)
10. Write `.deploy/bots/<slug>/docker-compose.workspace.yml` with:
    - `env_file:` entries pointing to workspace.env on each service
    - `volumes:` entries for workspace bind mounts on each service
    - Services: bot, bot-provider, bot-webhook, bot-worker

### Regeneration triggers

| Mutation | Affected bots |
|---|---|
| `workspace create` | None (empty members.txt) |
| `workspace remove` | All former members |
| `workspace add-bot` | The added bot |
| `workspace remove-bot` | The removed bot |

### Test strategy

- Generation with 0, 1, 2 workspace memberships
- Merging preserves existing BOT\_PROJECTS and BOT\_AGENT\_TAGS from .env
- Deduplication (same workspace name does not duplicate entries)
- `ro` workspace generates `inspect` file\_policy
- `rw` workspace generates `edit` file\_policy
- workspace.env removed when bot has no memberships
- Generated BOT\_PROJECTS round-trips through `_parse_projects()` in
  `app/config.py`
- Compose override YAML is valid

## Phase 3: Compose Wiring

### Files to modify

| File | Change |
|---|---|
| `scripts/lib/docker.sh` | Update `bot_compose()` and `bot_shared_compose()` |

### Changes to `bot_compose()`

After resolving env\_file and provider\_auth\_dir:
1. Check if `.deploy/bots/$slug/docker-compose.workspace.yml` exists
2. If so, add `-f .deploy/bots/$slug/docker-compose.workspace.yml` **after**
   the base compose file

The workspace compose override handles both concerns:
- **Volumes:** bind mounts for workspace directories (compose merges volume
  lists from override files)
- **`env_file:` entries:** workspace.env appended to each service's env\_file
  list, so `BOT_PROJECTS` and `BOT_AGENT_TAGS` reach the container process
  (not just YAML interpolation)

CLI `--env-file` is passed **only if workspace.env uses variables needed for
YAML interpolation** in the workspace compose override (e.g. path
substitution). For runtime config delivery to the container, the service-level
`env_file:` in the override is the required mechanism.

```bash
# bot_compose: local runtime (single process)
docker compose \
  --project-directory . \
  -p "octopus-${slug}" \
  -f infra/compose/docker-compose.yml \
  ${workspace_compose:+-f "$workspace_compose"} \
  --profile bot \
  --env-file "$env_file" \
  "$@"
```

```bash
# bot_shared_compose: shared runtime (webhook + workers)
# Workspace override must be LAST so volumes and env_file additions
# apply on top of both base and shared-runtime services.
docker compose \
  --project-directory . \
  -p "octopus-${slug}" \
  -f infra/compose/docker-compose.yml \
  -f infra/compose/docker-compose.shared.yml \
  ${workspace_compose:+-f "$workspace_compose"} \
  --profile bot \
  --env-file "$env_file" \
  "$@"
```

### Compose merge order

The explicit ordering matters:

| Position | File | Provides |
|---|---|---|
| 1 | `docker-compose.yml` | Base services, volumes, env\_file from `.env` |
| 2 | `docker-compose.shared.yml` | Webhook/worker split (shared mode only) |
| 3 | `docker-compose.workspace.yml` | Workspace volumes + workspace.env as `env_file` |

The workspace override is always last. Compose merges `env_file:` lists by
appending; within the merged list, duplicate keys resolve last-wins. Since
`.env` is loaded first and workspace.env is appended after, workspace values
win for `BOT_PROJECTS` and `BOT_AGENT_TAGS`.

### Permission requirements (documentation)

- Host directory must be readable (writable for rw mode) by uid 1000
- On macOS with Docker Desktop: automatic uid mapping, works out of the box
- On Linux: operator may need `chmod`/`chown` on the host directory
- `workspace create` prints a permission note

### Verification test (one-liner)

Before implementation, confirm `env_file:` merge semantics with current
Compose version:

```bash

# verify container sees the last-wins value.
```

### Test strategy

- `bot_compose` with and without workspace files
- `bot_shared_compose` with workspace: override is last in `-f` chain
- Workspace volumes appear in `docker compose config` output
- Container process sees `BOT_PROJECTS` from workspace.env (not just from

## Phase 4: Doctor Validation

### Files to modify

| File | Change |
|---|---|
| `octopus` | Add `workspace_verify()`, extend `cmd_doctor()` |

### `workspace_verify [workspace]`

For each workspace (or specified workspace):
1. Host path exists and is a directory
2. Host path is readable
3. Host path is writable (if mode=rw)
4. Warn if common secret patterns found at top level: `.env`, `credentials.json`,
   `.aws/`, `.ssh/`, `*.pem`, `*.key`
5. All member bots exist (have .env files)
6. workspace.env is up to date (regenerate and compare content only — ignore
   file mode differences to avoid flapping on permissions)
7. Generated BOT\_PROJECTS parses correctly

### BOT\_PROJECTS parse validation

```bash
validate_bot_projects_syntax() {
  local projects_value="$1"
  printf '%s' "$projects_value" | python3 -c "
import sys
raw = sys.stdin.read().strip()
if not raw:
    sys.exit(0)
for entry in raw.split(','):
    entry = entry.strip()
    if not entry or ':' not in entry:
        print(f'Invalid entry: {entry!r}', file=sys.stderr)
        sys.exit(1)
    name, rest = entry.split(':', 1)
    if not name.strip() or not rest.strip():
        print(f'Empty name or path: {entry!r}', file=sys.stderr)
        sys.exit(1)
sys.exit(0)
"
}
```

### Extend `cmd_doctor()`

After existing bot doctor logic, if the bot has workspace memberships:
- Verify workspace host paths exist
- Verify workspace.env is present and current

- Verify with valid workspace config
- Verify with missing host path

- Verify with stale workspace.env

No new code needed. Already handled in Phase 2.

existing tag flow is:

1. `BOT_AGENT_TAGS` in env → `config.agent_tags` via `app/config.py`
2. `AgentRuntime` includes `config.agent_tags` in heartbeat
3. `AgentDiscoveryQuery(tags=("workspace:shared-repo",))` finds matching agents
4. Registry UI shows tags in bot detail view
