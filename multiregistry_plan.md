# Multi-Registry Connections & Channel Bootstrap Plan

## Problem Statement

The current bot runtime has three structural limitations:

1. It assumes exactly one registry connection.
2. It hardwires channel dispatch through `conversation_channel_name()`
   and `create_channel_egress()`.
3. It treats Telegram as mandatory and uses a registry-issued
   `agent_id` as part of local conversation identity.

This plan removes those limitations without creating parallel queues,
parallel state machines, or half-pluggable paths.

There is no user-facing backward compatibility requirement. No env
file migration, no state file migration, no conversation ref shims.
Old user-facing code paths are deleted, not wrapped.

Database schema changes use normal migrations (fresh-install schema
updates + Postgres ALTER migrations) per the repo's existing store
parity discipline. This is standard DB engineering, not a backward
compatibility concern.

## Design Decisions

### D1: `registry_scope`, not `role`

`role` is already used as agent persona metadata (`AgentCard.role`
in `types.py`, discovery search in `store.py`). Reusing it for
registry permissions would destroy existing semantics.

New field: `registry_scope` — values: `channel`, `coordination`,
`full`. Used in config, env vars, state files, enrollment payload,
and agents table schema.

### D2: Stable local `bot_id` (runtime state only)

Conversation refs must not depend on any registry `agent_id`.
A stable local `bot_id` is generated once by the Python runtime and
persisted in the existing json state directory:

`<data_dir>/agent/bot_identity.json`

```json
{
  "bot_id": "a1b2c3d4e5f6...",
  "created_at": "2026-03-19T12:34:56Z"
}
```

Rules:
- `bot_id` is runtime state only. Generated by Python on first run.
- It is NOT written to the bot `.env` file. The CLI never reads it.
- `bot_identity(data_dir)` reads it from the data dir. It does not
  take `config` — `bot_id` is not config, it is persisted runtime state.
- Registry `agent_id` is per-registry-connection, not global.
- Telegram refs: `telegram:<bot_id>:<chat_id>`.
- `agent_identity()` replaced by `bot_identity(data_dir)`.

This follows the existing pattern: the Python runtime already manages
`registry_state.json` in the same directory. The CLI manages `.env`
files. They do not overlap.

### D3: Coordination stays registry-specific

Registry has two responsibilities: channel surface and coordination
control plane. We generalize the channel side. We keep coordination
registry-specific. Do not invent a generic `CoordinationPort` until
a second coordination backend exists.

### D4: Qualified conversation refs

Refs are channel-owned with explicit kind:

```
telegram:<bot_id>:<chat_id>
registry:<registry_id>:conversation:<conversation_id>
registry:<registry_id>:task:<routed_task_id>
```

No "anything not telegram is registry" fallback. Unknown refs are
rejected, not silently claimed.

### D5: Two delivery lanes, one queue

Channel deliveries: `channel_input`, `channel_action`.
Coordination deliveries: `routed_task`, `routed_result`.

All flow through the existing durable transport / work queue. No
second queue, no second worker FSM, no second transport store. The
registry filters poll results by `registry_scope` and delivery kind.

### D6: Telegram is optional

`validate_config()` requires at least one ingress-capable channel
(Telegram token set, OR at least one registry with
`registry_scope` in `channel` or `full`). A bot with no inbound
channels is invalid.

### D7: Honest pluggability

Adding a new channel requires:
1. A package under `app/channels/<name>/`
2. Channel-specific config parsing in `load_config()`
3. One registration entry in the startup module

It does not require edits to worker dispatch, delivery handlers,
composition helpers, or egress factory branches. Do not claim
zero-touch unless startup registration is fully externalized.

### D8: No second queue

All inbound work, regardless of source, flows through the existing
durable transport/work queue. No second queue. No second worker FSM.
Registry provenance (`registry_id`) travels with the delivery from
admission — not reverse-engineered from conversation refs later.

### D9: `channel_capabilities` derived from active channels

Current code hardcodes `channel_capabilities` from `agent_mode` in
`runtime.py:61`. With multi-registry and optional Telegram, this is
wrong.

New rule: enrollment/register payloads derive `channel_capabilities`
from the actually active channel bootstraps registered with the
dispatcher. The registry runtime queries the dispatcher for active
channel types via `dispatcher.active_channel_types()` and includes
only those in the agent card. This is a read-only query across the
dispatcher/runtime boundary — the dispatcher owns the channel type
data, the runtime reads it. The runtime does not own or cache
channel types.

```python
# Before (hardcoded):
channel_capabilities=("telegram", "registry") if agent_mode == "registry" else ("telegram",)

# After (derived from dispatcher — read-only query):
channel_capabilities=tuple(dispatcher.active_channel_types())
```

### D10: Slice ordering preserves a working repo

Old singleton config fields (`agent_registry_url`,
`agent_registry_enroll_token`) and the old state path
(`registry_state.json`) are NOT deleted until slice 12.

Two internal projections keep intermediate slices green:

**Config projection**: `load_config()` projects from
`agent_registries[0]` back to the old singleton fields
(`agent_registry_url`, `agent_registry_enroll_token`) so existing
runtime code that reads those fields continues to work.

**State projection**: The registry runtime (slice 4) dual-writes
the default connection's state to both `registries/default.json` AND
the old `registry_state.json` path. This keeps old consumers that
call `load_agent_runtime_state(data_dir)` (in `bridge.py`,
`runtime.py`, `registry/egress.py`) reading valid state until they
are migrated to connection-aware access in later slices.

Both projections are internal scaffolding, not user-facing shims.
Both are deleted in slice 12 along with all code that reads the old
fields/paths.

### D11: Coordination-only connections still need task egress

Coordination-only connections do not register a `ChannelBootstrap`
for conversations — they cannot receive `channel_input` deliveries.
But they CAN receive `routed_task` deliveries, and routed task
execution requires a `ChannelEgress` for progress/timeline/outcome.

Resolution: coordination-only connections register a **task channel**
(`Channel`, not `ChannelBootstrap`) that claims only the
`registry:<id>:task:` prefix. This channel builds a
`RegistryChannelEgress` scoped to the coordination connection's
client and state — enough to publish timeline events for task
execution. It does NOT claim `registry:<id>:conversation:` (that is
channel-only). It does not have ingress — the registry runtime's
poll loop owns inbound task delivery.

The task channel's descriptor has `contributes_channel_capability=False`
and `accepts_channel_input=False`, so it does NOT cause the bot to
advertise `"registry"` in `channel_capabilities`.

This means the dispatcher routes:
- `registry:prod:conversation:*` → conversation `Channel`
  (exists for channel/full connections)
- `registry:prod:task:*` → task `Channel` (exists for
  coordination/full connections)
- `telegram:*` → Telegram `ChannelBootstrap`

A coordination-only connection has a task channel but no conversation
bootstrap. A full connection has both. A channel-only connection has
a conversation bootstrap but no task channel (it does not receive
routed tasks).

### D12: Surface-specific logic audit

Bug class 14 from AGENTS.md says: *"If you add
`if conversation_ref.startswith(...)` in an orchestration file, you
have violated the port+factory rule."*

Current surface-specific logic outside the factory/composition pair:

- `worker.py:176-184` — `source == "telegram"` checks for chat_id
  and bot_instance requirements
- `work_admission.py:16` — `source == "registry"` → trust "trusted"
- `work_admission.py:37` — `source != "telegram"` → skip access check
- `delivery.py:192` — `conversation_channel_name()` for bot-required
- `workflows/execution/context.py:20-22` — branches on
  `metadata.agent_mode == "registry"` and
  `metadata.channel_name != "registry"`

The plan must resolve each:

**worker.py source checks**: These move into the channel's
`build_egress()`. The Telegram `ChannelBootstrap` validates
chat_id/bot internally. The dispatcher's `create_egress()` call
replaces the inline checks. The worker calls
`dispatcher.create_egress()` and does not inspect `source` for
channel-specific validation.

**work_admission.py trust tier**: `trust_tier_for_source()` currently
hardcodes `source == "registry"` → trusted. With multi-channel,
trust is a property of the channel, not a string check.
`ChannelDescriptor.trust_tier` carries it. Replace
`trust_tier_for_source(source, ...)` with
`trust_tier_for_ref(conversation_ref, ...)` that calls
`dispatcher.descriptor_for_ref(conversation_ref).trust_tier`. The
conversation ref is available at the admission boundary via
`InboundMessage.conversation_ref`. This is more precise — different
registries could have different trust tiers.

**delivery.py bot-required check**: Currently checks
`conversation_channel_name() == "telegram"`. Replace with
`dispatcher.channel_type_for_ref(parent_conversation_id) == "telegram"`.
This is a dispatcher query, not a hardwired branch — it goes through
the registered prefix map.

**context.py agent_mode/channel_name checks**: Currently branches on
`metadata.agent_mode == "registry"` and
`metadata.channel_name != "registry"`. Replace with channel-agnostic
logic: query `dispatcher.descriptor_for_ref(conversation_ref)` to
get `supports_conversation_binding` and `supports_timeline`. If
`conversation_ref` is set and the descriptor says the channel
supports binding, build the execution channel context with timeline
callback. Do not branch on `agent_mode` or `channel_name` strings.

**API threading for context.py**: The current function signature
receives `ExecutionChannelMetadata` which contains `agent_mode` and
`channel_name`. These are the fields being eliminated. The function
gains a `descriptor: ChannelDescriptor | None` parameter, populated
by the caller (worker dispatch) via `dispatcher.descriptor_for_ref()`.
The `agent_mode` and `channel_name` fields are removed from
`ExecutionChannelMetadata` — the descriptor replaces them. The
metadata retains `message_conversation_ref`, `routed_task_id`, and
`chat_id`.

Slice 6 must resolve all five locations. The implementation prompt
must list them explicitly.

### D13: Test policy

Full test suite runs after every slice. No exceptions. The
implementation prompt's "targeted tests + full suite on storage
slices" was inconsistent with the exit gate requiring every slice
green. The exit gate wins.

### 4. Channel abstraction

#### Type hierarchy

```
Channel               — owns a ref prefix, builds egress
  ChannelBootstrap    — a Channel that also runs ingress
ChannelIngress        — an independent ingress runner (built by ChannelBootstrap)
ChannelEgress         — already exists in app/ports/egress.py (unchanged)
ChannelDescriptor     — metadata about a channel instance
```

`Channel` is the root contract. Every ref in the system has exactly
one `Channel` owner. Some channels also have ingress (they are
`ChannelBootstrap`). Some are egress-only (they are just `Channel`).

#### Ports (`app/ports/channel.py`)

```python
@dataclass(frozen=True)
class ChannelDescriptor:
    channel_type: str                       # "telegram", "registry", "slack"
    display_name: str
    supports_multiple: bool
    requires_polling: bool
    trust_tier: str = "untrusted"           # "trusted" for registry, "untrusted" for telegram
    contributes_channel_capability: bool = True   # False for task-only channels
    accepts_channel_input: bool = True            # False for task-only channels
    supports_conversation_binding: bool = True    # True for channels that bind conversations to registries
    supports_timeline: bool = True                # True for channels that publish timeline events


class Channel(ABC):
    """Owns a ref prefix and builds egress for it."""

    @property
    @abstractmethod
    def channel_id(self) -> str:
        """Unique runtime identifier (e.g. 'telegram', 'registry:prod:conversation')."""
        ...

    @property
    @abstractmethod
    def descriptor(self) -> ChannelDescriptor:
        """Static channel metadata."""
        ...

    @abstractmethod
    def ref_prefix(self) -> str:
        """The ref prefix this channel owns (e.g. 'telegram:', 'registry:prod:task:')."""
        ...

    @abstractmethod
    def build_egress(self, *, conversation_ref: str, config: Any, **kw) -> ChannelEgress:
        ...


class ChannelBootstrap(Channel):
    """A Channel that also runs its own ingress loop."""

    @abstractmethod
    def build_ingress(self, *, config: Any, delivery_handler: Callable) -> ChannelIngress:
        ...


class ChannelIngress(ABC):
    """An independent ingress runner."""

    @property
    @abstractmethod
    def channel_id(self) -> str: ...

    @property
    @abstractmethod
    def descriptor(self) -> ChannelDescriptor: ...

    @abstractmethod
    async def start(self, *, stop_event: asyncio.Event) -> None: ...

    @abstractmethod
    async def stop(self) -> None: ...

    @abstractmethod
    async def health_check(self) -> dict[str, Any]: ...
```

Keep existing `ChannelEgress` ABC in `app/ports/egress.py` unchanged.

#### Concrete channel instances

| Instance | Type | `ref_prefix()` | `contributes_channel_capability` | `accepts_channel_input` | `supports_conversation_binding` | `supports_timeline` |
|---|---|---|---|---|---|---|
| Telegram | `ChannelBootstrap` | `telegram:` | `True` | `True` | `True` | `True` |
| Registry conversations | `Channel` | `registry:<id>:conversation:` | `True` | `True` | `True` | `True` |
| Registry tasks | `Channel` | `registry:<id>:task:` | `False` | `False` | `False` | `True` |

Both registry channel types are `Channel` (egress-only), not
`ChannelBootstrap`. Their inbound delivery is owned by the registry
runtime's poll loop. No dummy `build_ingress()`. No lying about
ownership. Only Telegram is a `ChannelBootstrap` — it is the only
channel that owns an independent ingress loop (PTB poll/webhook).

`contributes_channel_capability` controls whether a channel is
advertised in `channel_capabilities`. Registry conversations have
it `True` (they accept conversations). Registry tasks have it
`False` (they are egress-only for task progress). This is a
descriptor flag, not a type hierarchy distinction.

#### Dispatcher (`app/runtime/channel_dispatcher.py`)

Prefix-map dispatch. O(1) egress routing. Conflict detection at
registration time. No iteration, no fallback, no "anything not
telegram is registry."

```python
class ChannelDispatcher:
    register(channel: Channel) -> None
        # Stores ref_prefix → channel mapping
        # Rejects conflicting prefixes at registration

    create_egress(conversation_ref, *, config, **kw) -> ChannelEgress
        # Longest-prefix match against registered channels
        # Raises ValueError for unknown refs

    channel_type_for_ref(conversation_ref) -> str | None
        # Returns channel_type without building egress

    active_channel_types() -> list[str]
        # Returns channel_types where descriptor.contributes_channel_capability is True
        # Deduplicated. Used by registry runtime for channel_capabilities.
        # Read-only query — dispatcher owns the data, runtime reads it.

    build_all_ingresses(*, config, delivery_handler) -> dict[str, ChannelIngress]
        # Iterates registered channels. For each ChannelBootstrap, calls
        # build_ingress(). Stores the result. Skips plain Channel instances
        # (they have no ingress). Returns channel_id → ChannelIngress map.

    async start_all_ingresses(*, stop_event) -> None
        # Creates an asyncio.Task per built ingress, calling
        # ingress.start(stop_event=stop_event). All run concurrently.
        # Called from main.py post_init.

    async stop_all_ingresses() -> None
        # Calls ingress.stop() on each running ingress. Awaits all.
        # Called from main.py shutdown.

    descriptor_for_ref(conversation_ref) -> ChannelDescriptor | None
        # Returns descriptor for a conversation ref's channel.
        # Used by context.py to query supports_conversation_binding
        # and supports_timeline without building egress.

    get_ingress(channel_id) -> ChannelIngress | None
        # Only for ChannelBootstrap instances that built an ingress
```

The dispatcher registers `Channel` instances (the base). It does NOT
own registry clients, scopes, or mirroring. It owns ref → channel
routing, ingress lifecycle, and egress creation.

#### Ingress startup wiring in `main.py`

The current startup seam in `main.py` (lines 249-301) hardwires
`build_bootstrap()` for Telegram and `start_agent_runtime_task()`
for registry. The new model replaces both with the dispatcher:

```python
# In main.py post_init:
dispatcher = ChannelDispatcher()

# Register channels (Telegram, registry conversations, registry tasks)
# ... (done during config/connection setup)

# Build all ingresses from registered ChannelBootstrap instances
ingresses = dispatcher.build_all_ingresses(
    config=config,
    delivery_handler=delivery_handler,
)

# Start all ingresses as concurrent tasks
await dispatcher.start_all_ingresses(stop_event=stop_event)

# On shutdown:
await dispatcher.stop_all_ingresses()
```

This replaces the direct `build_bootstrap()` call for Telegram.
Only `ChannelBootstrap` instances get ingress built and started —
currently that is Telegram only. Registry conversation and task
channels are `Channel` (egress-only) and have no ingress. All
registry polling is owned by the `RegistryRuntime`.

### 5. Registry runtime (`app/agents/registry_runtime.py`)

Holds per-connection lifecycle. Owns registry clients and mirroring
fan-out. NOT the dispatcher's concern.

```python
class RegistryRuntime:
    # One client per connection
    # One RegistryConnectionState per connection
    # One poll cursor per connection (single owner — no cursor sharing)
    # One sync loop per connection: heartbeat + enrollment + poll
    #   (poll returns only kinds allowed by that connection's registry_scope)
    # Fan-out discovery across coordination/full connections
    # Route delegated task submission to correct registry
    # Route status/result updates through originating registry

    def __init__(self, registries, dispatcher, delivery_handler):
        # delivery_handler: called for each polled delivery
        # The runtime annotates each delivery with registry_id before
        # passing to the handler. The handler does not need to know
        # which registry the delivery came from — it receives it as
        # metadata.

    async start(*, stop_event: asyncio.Event) -> None
        # Starts all connection sync loops as concurrent tasks.
        # Each loop wraps an AgentRuntime instance scoped to one connection.
        # AgentRuntime.poll_once() is parameterized with a kind filter
        # derived from the connection's registry_scope (extension, not
        # rewrite — adds a parameter to the existing method).
        # Called from main.py post_init, AFTER dispatcher.start_all_ingresses().

    async stop() -> None
        # Stops all connection sync loops. Deregisters from each registry.
        # Called from main.py shutdown, BEFORE dispatcher.stop_all_ingresses().

    clients_for_mirroring() -> list[tuple[str, AgentRegistryClient]]
        # Returns (registry_id, client) for channel/full connections
        # Used by Telegram mirroring fan-out

    channel_capabilities() -> tuple[str, ...]
        # Derived from active dispatcher channel types
        # Used in enrollment/register agent card payloads
```

#### Startup/shutdown ordering in `main.py`

```python
# 1. Build dispatcher, register Telegram
dispatcher = ChannelDispatcher()
if config.telegram_token:
    dispatcher.register(TelegramChannelBootstrap(config, provider))

# 2. Build registry runtime (owns clients, state, sync loops)
#    Channel registration happens via registry_runtime.register_channels()
#    after channel classes are available (slice 5+).
registry_runtime = RegistryRuntime(config.agent_registries, dispatcher, delivery_handler)
registry_runtime.register_channels()  # creates and registers Channel instances with dispatcher

# 3. Build and start channel ingresses (Telegram only — registry
#    channels are Channel, not ChannelBootstrap, so no ingress)
dispatcher.build_all_ingresses(config=config, delivery_handler=delivery_handler)
await dispatcher.start_all_ingresses(stop_event=stop_event)

# 4. Start registry runtime sync loops (heartbeat, enrollment, poll)
await registry_runtime.start(stop_event=stop_event)

# On shutdown (reverse order):
# 5. Stop registry runtime (stop polling, deregister from registries)
await registry_runtime.stop()
# 6. Stop channel ingresses (stop Telegram)
await dispatcher.stop_all_ingresses()
```

The dispatcher manages `ChannelBootstrap` ingresses (Telegram only).
The registry runtime manages all registry sync loops (heartbeat,
enrollment, poll). Registry channels are `Channel` (egress-only) —
no ingress to manage.

#### Poll cursor ownership

Each registry connection has ONE poll cursor and ONE poll loop,
both owned by the `RegistryRuntime`. The server-side `poll()`
endpoint filters deliveries by `registry_scope`:
- `channel` scope: returns `channel_input`, `channel_action` only
- `coordination` scope: returns `routed_task`, `routed_result` only
- `full` scope: returns all kinds

There is no `RegistryConversationIngress`. Registry conversations
are a `Channel` (egress-only), not a `ChannelBootstrap`. They do
not have ingress. All registry polling is in the `RegistryRuntime`,
which calls the shared `delivery_handler` for each polled delivery.
The handler routes deliveries to the work queue regardless of kind.

This avoids the cursor-race problem: if channel-lane and
coordination-lane had separate poll loops sharing one cursor, they
would skip each other's deliveries. One cursor, one poll loop, one
owner. No fake no-op ingress.

Startup order: dispatcher ingresses first (so Telegram is reachable
and registry conversation egress is routable), then registry runtime
(heartbeat, enrollment, poll for all lanes).
Shutdown order: registry runtime first (deregister, stop polling),
then dispatcher ingresses (stop Telegram).

The `RegistryRuntime` creates and registers channels with the
dispatcher via `register_channels()` (called separately from
`__init__`, so it can be deferred until slice 5 when channel classes
exist). The runtime owns the per-connection clients and state that
the channels need:

- Channel-capable connections (`channel`, `full`): the runtime creates
  a `RegistryConversationChannel` (`Channel`, not `ChannelBootstrap`)
  and registers it with the dispatcher. It claims
  `registry:<id>:conversation:` and builds egress. It has
  `contributes_channel_capability=True`. No ingress — the runtime
  owns all registry polling.

- Coordination-capable connections (`coordination`, `full`): the
  runtime creates a `RegistryTaskChannel` (`Channel`, not
  `ChannelBootstrap`) and registers it with the dispatcher. It claims
  `registry:<id>:task:` and builds egress only. Its descriptor has
  `contributes_channel_capability=False` (see D11).

- Full connections register both.

- Coordination-only connections run their sync loop through the
  registry runtime but have no conversation channel — only task egress.

This means `main.py` does not need to know about connection scopes
or create registry channels directly. It creates the `RegistryRuntime`
and calls `register_channels()`. The runtime handles the details
of which channels to create based on each connection's scope.

### 6. Telegram mirroring fan-out

Telegram conversations are mirrored (bindings + timeline events) to
all registries where the bot is attached as `channel` or `full`.
Coordination-only connections receive no Telegram mirrors.

Bridge fan-out functions use the registry runtime, not the dispatcher:

```python
async def bind_conversation_to_registries(
    registry_runtime: RegistryRuntime,
    *, conversation_ref, title, origin_channel, external_id,
) -> None:
    for registry_id, client in registry_runtime.clients_for_mirroring():
        try:
            await client.sync_binding(...)
        except RegistryClientError:
            log.debug("Registry %s bind failed for %s", registry_id, conversation_ref)

async def publish_timeline_to_registries(
    registry_runtime: RegistryRuntime,
    *, conversation_ref, kind, title, body, ...
) -> None:
    for registry_id, client in registry_runtime.clients_for_mirroring():
        try:
            await client.publish_timeline([event])
        except RegistryClientError:
            log.debug("Registry %s timeline failed for %s", registry_id, conversation_ref)
```

`TelegramChannelEgress` calls these fan-out functions via the
registry runtime.
`RegistryChannelEgress` talks to its own specific registry client
(no fan-out — it knows its `registry_id`).

- Delegation submits through that result's `registry_id`.

### 8. Inbound dispatch model

All inbound work flows through the existing durable transport/work
queue. Registry provenance is carried as metadata:

- `InboundMessage` and `InboundAction` gain `registry_id` field.
- Delivery handler tags deliveries with `registry_id` at admission.
- Worker does not need to know `registry_id` — egress is created from
  `conversation_ref` via the dispatcher, which routes to the correct

## Hard Rules

1. **Do not overload `role`.** Use `registry_scope`. `role` is
   persona metadata. Different field, different column, different
   env var suffix.

2. **`bot_id` is runtime state only.** Persisted in
   `bot_identity.json` by Python. Not in `.env`. Not touched by CLI.
   Not derived from registry `agent_id`.

3. **No second queue, no second worker FSM, no second transport
   store.** All ingress ends in the existing durable transport path.

4. **No hardwired channel dispatch after slice 6.** All egress
   creation goes through `ChannelDispatcher.create_egress()` with
   prefix-map routing. Slice 5 registers bootstraps; slice 6 deletes
   the old factory.

5. **One connection = one client, one state file, one poll cursor,
   one failure domain.**

6. **Registry store parity.** Every store change in one slice:
   `store_base.py`, `store.py`, `store_postgres.py`, migration,
   contract test.

7. **`AgentRuntime` is wrapped, not rewritten.** One extension is
   required: `poll_once()` gains an optional `kind_filter` parameter
   so the registry runtime can request only the kinds allowed by
   that connection's `registry_scope`. This is a backward-compatible
   parameter addition, not a rewrite of the poll logic.

8. **Qualified refs with kind.** Unknown refs rejected. No fallback.

9. **Registry provenance travels with the delivery.** `registry_id`
   tagged at admission. Not reverse-engineered from conversation refs.

10. **Human-facing errors name the registry.** "Could not reach
    registry 'prod' at https://..." — not "registry unreachable."

11. **`channel_capabilities` derived from active channels.** Not
    hardcoded from `agent_mode`. Query the dispatcher for active
    channel types.

12. **Dispatcher owns routing and Telegram ingress. Registry runtime
    owns all registry lifecycle.** The dispatcher holds the ref →
    channel prefix map and manages Telegram ingress. The
    `RegistryRuntime` creates registry channels via
    `register_channels()` and registers them with the dispatcher for
    egress routing. The dispatcher does not hold registry clients,
    scopes, or poll loops.

13. **Each slice leaves the repo green.** Full test suite after every
    slice. Config projection and state dual-write keep old consumers
    alive until slice 12.

14. **Do not claim pluggability the code does not provide.**

15. **Coordination-only still needs task egress.** Register a
    task `Channel` (not `ChannelBootstrap`) for `registry:<id>:task:`
    prefix. Routed task execution depends on `ChannelEgress` for
    progress/outcome. Task channel descriptor has
    `contributes_channel_capability=False`.

16. **No surface-specific branching in orchestration code.** Per bug
    class 14, `worker_dispatch`, `delivery.py`, `work_admission.py`
    must not contain `if source == "telegram"` or
    `if ref.startswith(...)`. Channel-specific validation goes in the
    bootstrap. Trust tier goes in `ChannelDescriptor`.
